State Machines in Controllers
Getting Started > State Machine Graphs > Using A State Machine > State Machines in Controllers

Controllers are essentially a group of command implementations that modify's the data of an element.  But they are also a great place to initialize different things on a view-model that you want to be unique to the controller rather than the view-model itself.  State Machines are a great example of this.  Computed properties, and commands are a great way to create state machines, but these, by default, haven't any sense of time.  Traditionally this would require a view to execute a command at the right time. 

It's still a great idea to add commands even if that command might be invoked automatically from the controller.

With the power of UniRX and it's Timer and Interval methods we can create time based state machines that are generic and can work in other environments as well. 

For Example lets look at one of the examples with uFrame.  In the controller we override the generated base method Initialize{ElementName}({ElementName}ViewModel vm)

InitializeFPSWeapon
Copy Code 
    public override void InitializeFPSWeapon(FPSWeaponViewModel fpsWeapon)
    {
        fpsWeapon.StateProperty
            .Where(p => p is Reloading) // Subscribe only to when the state is changed to reloading
            .Subscribe(r =>
            {
                // When we've entered the reloading state create a timer that moves it to finished reloading
                Observable.Timer(TimeSpan.FromSeconds(fpsWeapon.ReloadTime))
                    .Subscribe(l =>
                    {
                        fpsWeapon.Ammo = fpsWeapon.RoundSize;
                        ExecuteCommand(fpsWeapon.FinishedReloading);
                    });
            }).DisposeWith(fpsWeapon); // Make sure this is disposed with the weapon
    }

In the above code we are subscribing to any state changes, when the state changes to reloading we create a timer.  When the timer completes we reset the ammo, and execute the command "FinishedReloading".  This comletely elimnates any need for a view to invoke a command, this is more of an internal operation and doesn't have much to do with how something is displayed.
