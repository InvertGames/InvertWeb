The Anatomy Of UFrame

So now that we've talked about the general concept of uFrame.  We need to go over how this is all put together in uFrame.

Scene Loading/Booting

The first thing to load in uFrame is the GameManager.  This happens because of its predefined "Script Execution Order".  When the GameManager loads it makes sure that it is the first instance to load.  If it's not the first instance then it will transfer the properties of itself to the current instance and then destroy itself.

Then the GameManager will begin by calling the "Start" SceneManger's setup method that is specified in the "Start" property located on the GameManger's inspector options.  Before we go much further lets examine this setup method.

The Setup Method

This method could be defined as the scene bootstraper.  It's responsibility is to register any dependencies with the dependency container that comes with uFrame.

1.public override void Setup()
2.{
3.    base.Setup();
4.         
5.}


At first glance, it looks like an empty method that calls its base class's setup method.  They key here is in the base call because uFrame generates the basic bootstrapper for you.  To fully understand how its registering dependencies lets take a look at simple example of a generated setup method.

1.public override void Setup() {
2.     // Register every controller defined in the subsystems that are connected to the scene manager in the designer
3.     this.MyElementController = new MyElementController() { Container = Container, Context = Context };
4.     Container.RegisterInstance(this.MyElementController, false);
5.     Container.InjectAll(); // Inject all the controllers at once
6.}

There is more to this method than displayed but we'll talk about that in a second.  The key points to notice in the above example is line 3 and 4 where it's registering a single instance of a controller.  This means that when we call

1.Container.Resolve<MyElementController>();

it will return the same instance of the controller we registered in the Setup method.  All this means is that the controller, in its own form is a Singleton.  

We register these controllers in the setup method because it is important that they are ready to be used as unity loads the views and GameObjects in the scene.


Ok and now for the next part of the setup method.  The next part is where we register every single instance Element's View-Model so that they will behave like a Singleton just like all of the controllers do.

1.public override void Setup() {
2.     // ... the other stuff from above
3.     Container.RegisterInstance<MyElementViewModel>(SetupViewModel<MyElementViewModel>(MyElementController,"MyElement"));
4.}

The key part to note here is that we are calling "SetupViewModel".  This is simply a helper method for creating Single-Instance View-Models.  This method will create the ViewModel and assign it the appropriate controller passed in the first argument if it doesn't already exist.

Another to thing to note is that when we register instances with this container.  It will inject the instances with any dependencies they require.  This is done with the "Inject" attribute defined on a property or field. uFrame automatically generates controller properties inside of each controller based on the relationships to its element.  For example:

1.public abstract class CheckerBoardControllerBase : Controller {
2.     
3.    [Inject] public CheckerController CheckerController {get;set;}
4.    [Inject] public CheckerPlateController CheckerPlateController {get;set;}
5.    [Inject] public CheckersGameController CheckersGameController {get;set;}
6.     
7.    //... other stuff removed for clarity
8.}

This is a generated base controller for the CheckerBoard element in the included checkers game example. 

So when we register the instance of this controller.  The container will scan it looking for properties with inject attribute.  If it finds a property with the inject attribute it will "Resolve" a registered instance based on the type of the property.

You could easily create your own properties and mark them with the [Inject] attribute and they will work the same way controllers do as long as you have registered it in the Setup method before the base.Setup() line is executed.

This basically sums up how the purpose of the Setup method and how it initializes controllers and single instance view-models on scene load.  Lets move on to what happens next after the setup method is invoked. 

The Load Method

After the setup method is called by the game-manager.  It will then call the load method coroutine that is also on the SceneManager. This is a great place to create any root views (if any) that need to be dynamically instantiated when the scene loads.  This is also a great place to load a serialized state of a game. (See: Scene State Saving & Loading )

During this load method coroutines execution the Scene is also loading.  When each view loads it will call RequestViewModel on the SceneManager.  So lets take a look at whats going on here.


The RequestViewModel Method

This method provides view-models to all of the views in a scene on load.  The view will pass itself, the controller it should use, as well as the identifier for the view-model as well. 

Then the method will take the identifier, search for it in the context first, then in the dependency container, and if nothing is found it will create a new one.  This is why when you set the same identifier or "ResolveName" on two different views they will get the same instance of the view-model and thus both re-act as the view-model properties and collections change.

 This method will always ensure that the view-model exists in the context.  This allows any view-model to be located very easily by its identifier. 

As a last and final step it will check if the view is marked as "Save & Load".  If it is it will add it to a separated "PersitableViews" dictionary that can easily be iterated and serialized for saving and loading a scenes state.

In the last step before returning the requested view-model it will invoke "InitializeData" on the view which simply maps the properties specified in the view's inspector directly to the view-model.  Then it will call the Initialize{ElementName} on the controller passing the reference to the view-model.  This gives the controller the ultimate initialization power.  Anything applied in this method will override anything that has been set in the inspector.

After a view has received it's view-model from the scene manager, the Bind method on the view will be called which will appropriately wire up the various binding methods to the property and collection change events.

Note: When using uFrame InstantiateView method you can pass in a ViewModel directly in which it will skip this method entirely.

The OnLoaded Method

So after all of this has been completed the OnLoaded method will be called.  There isn't much to note about its implementation because it doesn't have one.  It is simply for your own use to know when everything has been properly loaded and the game is ready to begin.
